  <action>
    Em app/channels/client_calendar_channel.rb, substituir o corpo do método subscribed:

    ANTES (linhas 2-5):
    ```
    def subscribed
      reject unless current_client
      stream_for current_client
    end
    ```

    DEPOIS:
    ```
    def subscribed
      return reject unless current_client
      stream_for current_client
    end
    ```

    A única mudança é `reject unless` → `return reject unless`. O `return` garante que quando current_client é nil, o método retorna imediatamente após invocar reject — stream_for nunca é chamado com nil.

    Por que isso importa: ActionCable::Channel::Base#reject apenas seta @reject_subscription = true (linha 270-272 de actioncable-8.1.3). Sem return, a execução continua para stream_for(nil). stream_for(nil) chama broadcasting_for(nil) que com Array(nil) = [] produz o broadcasting name "client_calendar_channel" — uma stream global não-scoped. Toda conexão rejeitada fica inscrita nessa stream parasita consumindo recursos.

    O teste existente `assert subscription.rejected?` continua passando porque verifica o flag boolean, que a mudança não afeta.
  </action>
