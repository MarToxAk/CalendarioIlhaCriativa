# Phase 3 — UI Design Contract
# Art Management

> Visual and interaction contract for the art management screens (admin).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Inheritance

All design tokens (colors, typography, spacing, borders, shadows, transitions) are
**inherited from Phase 1 — see `.planning/phases/01-data-foundation-security/01-UI-SPEC.md`**
and Phase 2 patterns — see `.planning/phases/02-admin-auth-client-management/02-UI-SPEC.md`.

This document defines only what is new or additive to Phases 1–2.

---

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | none | Inherited Phase 1 |
| Preset | not applicable | Inherited Phase 1 |

| Icon library | Heroicons (inline SVG) | Inherited Phase 1 |
| Font | Inter via Google Fonts | Inherited Phase 1 |

---

## Focal Point

Focal point: the arte form is the primary creation surface; "Salvar arte" is the primary action focal point. On the global arte list, the arte table with thumbnail column is the visual anchor; "+ Nova arte" is the list-level CTA.

---

## Spacing Scale

Inherited from Phase 1 — 8-point scale: 4, 8, 16, 24, 32, 48, 64px.

Phase 3 exceptions:
- Thumbnail size in list: 60×60px (not on the 8-point scale, but consistent with D-07)
- Radio toggle (upload/link) label padding: 12px vertical / 16px horizontal (inside toggle pill)
- Platform icon: 16×16px inline in table, 20×20px on show page
- Touch target for radio toggle: 44px minimum height (accessibility requirement)

---

Maximum 4 sizes in use (12, 14, 24 — established set from Phase 1/2).

### Screen 4 — `/admin/artes/:id/edit` — Editar Arte

Estrutura idêntica ao formulário de nova arte, com as seguintes diferenças:
- Topbar título: "Editar arte"
- Link breadcrumb: "← Voltar para [Título da arte]"
- Campos pré-preenchidos com valores atuais
- Campo Cliente: `disabled` (não editável — mudar cliente de arte existente é operação de dados, fora do escopo)
  - Hint abaixo: "Para mudar o cliente, exclua esta arte e crie uma nova."
- Se arte tem upload existente: área de upload mostra preview do arquivo atual com opção "Substituir arquivo"
- Se arte tem link externo: painel de link externo ativo com valor preenchido
- Botão de submit: "Salvar alterações" (Primary Admin)
- Botão secundário: "Voltar sem salvar" → redireciona para `/admin/artes/:id`
- Se arte não está com status `pending`: formulário exibido como readonly (ver Interaction Pattern 3 — Bloqueio de edição)

## Interaction Patterns

### Padrão 1 — Toggle Upload / Link Externo (Stimulus `media_type_toggle`)

**Trigger:** Clicar em "Upload de arquivo" ou "Link externo" no toggle pill

**Comportamento:**
1. Stimulus controller `media_type_toggle` recebe o click
2. Oculta o painel não selecionado com `hidden` class (não remove do DOM)
3. Exibe o painel selecionado
4. Ao ocultar "Upload": limpa o arquivo selecionado (se houver) — evita envio acidental de ambos
5. Ao ocultar "Link externo": limpa o campo de URL — mesma razão
6. Tab ativo: `bg-white shadow-sm`, tab inativo: sem fundo, cor text-slate-600
7. Comportamento idêntico ao `password_toggle_controller` em termos de troca de classes via `data-media-type-toggle-*` targets

**ARIA do toggle:**
```html
<div role="tablist" aria-label="Tipo de mídia">
  <button role="tab" aria-selected="true"  aria-controls="panel-upload"  id="tab-upload">Upload de arquivo</button>
  <button role="tab" aria-selected="false" aria-controls="panel-link"    id="tab-link">Link externo</button>
</div>
<div id="panel-upload" role="tabpanel" aria-labelledby="tab-upload">...</div>
<div id="panel-link"   role="tabpanel" aria-labelledby="tab-link" hidden>...</div>
```

---

### Padrão 2 — Exclusão de Arte (modal de confirmação)

**Trigger:** Clicar em "Excluir arte" no topo do detalhe ou no menu "…" da lista

**Comportamento:**
1. Abre modal de confirmação (reutiliza `_confirm_modal` de Phase 2)
2. Admin confirma
3. DELETE para `admin_arte_path(@arte)`
4. Redirect para `/admin/artes` com flash de sucesso

**Modal de confirmação — Excluir arte:**
```
┌─────────────────────────────────────────────┐
│  Excluir arte                          [✕]  │
│  ─────────────────────────────────────────  │
│  Excluir "[Título da arte]" é uma ação       │
│  permanente. Esta arte não poderá ser        │
│  recuperada.                                 │
│                                             │
│    [Manter arte]   [Excluir arte]            │
└─────────────────────────────────────────────┘
```

- Botão "Manter arte": Secondary — fecha o modal
- Botão "Excluir arte": Danger (bg #EE3537) — executa DELETE

---

### Padrão 3 — Bloqueio de Edição / Exclusão (status != pending)

**Cenário:** Arte com status `approved`, `change_requested` ou `revised`.

**Comportamento na tela de detalhe:**
- Botões "Editar arte" e "Excluir arte" não aparecem na barra de ações
- Banner de aviso abaixo da barra de ações:
  ```
  ┌───────────────────────────────────────────────────────┐
  │  ⓘ  Esta arte não pode ser editada porque já recebeu  │
  │     uma resposta do cliente.                          │
  └───────────────────────────────────────────────────────┘
  ```
  - Estilo: `bg-[#FFFBEB] border border-[#F59E0B]/30 rounded-lg px-4 py-3 flex gap-2 items-start text-sm text-amber-800`
  - Ícone: `information-circle` Heroicons 16px, `text-[#F59E0B] flex-shrink-0 mt-0.5`

**Comportamento no controller (não visível na UI mas documenta o contrato):**
- `edit` e `update` retornam redirect com flash de erro se `!arte.pending?`
- Flash: "Esta arte não pode ser editada porque já recebeu uma resposta do cliente."
- `destroy` retorna redirect com flash de erro se arte tem approval_response ou `!arte.pending?`
- Flash: "Esta arte não pode ser excluída porque já recebeu uma resposta do cliente."

**Comportamento no menu "…" da lista:**
- Se `arte.pending?`: opções "Ver arte" / "Editar arte" / "Excluir arte"
- Se `!arte.pending?`: opções apenas "Ver arte" (sem Editar nem Excluir)

---

### Padrão 4 — Feedback de Validação no Formulário

**Trigger:** Submit com campos inválidos

**Comportamento:**
1. Rails renderiza o formulário de volta com `@arte.errors`
2. Cada campo com erro: borda vermelha `border-[#EE3537]` + mensagem abaixo
3. Banner de erro no topo do card:
   ```
   ┌────────────────────────────────────────────────────┐
   │  ✕  Corrija os erros abaixo antes de salvar.       │
   └────────────────────────────────────────────────────┘
   ```
   - Estilo: `bg-[#FEF2F2] border border-[#EE3537]/30 rounded-lg px-4 py-3 flex gap-2 items-center text-sm text-[#EE3537]`
