## Warnings

### WR-01: Upload field div visibility condition does not match radio pre-check logic

**File:** `app/views/admin/artes/_form.html.erb:37,41`

**Issue:** The upload radio button is pre-checked when `arte.media_file.attached? || arte.external_url.blank?` (line 37). For a new `Arte` object (`media_file.attached? = false`, `external_url = nil`) this evaluates to `true`, so the upload radio is selected. However, the upload field `<div>` on line 41 computes its CSS class as `arte.media_file.attached? ? 'mb-4' : 'mb-4 hidden'`, which evaluates to `'mb-4 hidden'` for the same new Arte — hiding the file input. The two expressions are inconsistent.

The Stimulus controller's `connect()` calls `toggleFields()` immediately, which detects the checked radio and removes the `hidden` class. So with JS enabled, users never see a broken form. But:

1. There is a brief flash of a hidden file input before Stimulus connects, which can cause visible layout jank on slower connections or under Turbo navigation.
2. With JavaScript disabled (or if the Stimulus controller fails to load), the upload radio appears selected while the file input is hidden. The user cannot upload a file, and submitting the form will trigger the `media_source_present` validation error with no way to resolve it from the UI.
3. The same mismatch applies to the `:new` render after a failed create attempt with `media_source=upload`.

**Fix:** Align the div's visibility condition with the radio's checked condition:

```erb
<%# line 41 — was: arte.media_file.attached? ? 'mb-4' : 'mb-4 hidden' %>
<div class="<%= (arte.media_file.attached? || arte.external_url.blank?) ? 'mb-4' : 'mb-4 hidden' %>"
     data-media-type-toggle-target="uploadField">
```

This makes the SSR-rendered HTML consistent with the pre-checked radio state, eliminating the flash and the no-JS failure.

## Info

### IN-01: Test fixture passes `:status` param that is silently filtered by strong parameters

**File:** `test/controllers/admin/artes_controller_test.rb:28`

**Issue:** The `"should create arte"` test passes `status: :pending` inside the `arte:` params hash. `arte_params` does not permit `:status`, so Rails silently discards it. The test passes only because `pending` (0) is the column default — the assertion is vacuously correct. If the default status were changed, this test would continue to pass without actually verifying the intended behaviour.

**Fix:** Remove `:status` from the params hash in this test. If verifying default status on creation is desired, assert it explicitly:

```ruby
post admin_artes_url, params: { arte: { client_id: @client.id, scheduled_on: Date.current,
  platform: :instagram, media_type: :image, title: "Nova Arte", caption: "Teste",
  approval_deadline: Date.current + 5, external_url: "https://drive.google.com/file/novo" } }
assert Arte.last.pending?, "newly created arte should have default pending status"
```

### IN-02: No tests for CR-01 guard on `new`, `create`, `update`, and `destroy` actions with invalid `client_id`

**File:** `test/controllers/admin/artes_controller_test.rb:141-165`

**Issue:** The CR-01 test coverage only exercises `index` and `edit` with an invalid `client_id`. The `set_client` before-action also runs for `new`, `create`, `show`, `update`, `destroy`, and `mark_revised`. Any of these actions with `client_id: 99999` should redirect to `admin_artes_url` with the alert. No tests verify this for the write paths (`create`, `update`, `destroy`).

**Fix:** Add cases for the remaining actions, at minimum for the mutating paths:

```ruby
test "CR-01: new com client_id invalido redireciona" do
  get new_admin_arte_url(client_id: 99999)
  assert_redirected_to admin_artes_url
end

test "CR-01: destroy com client_id invalido redireciona" do
  delete admin_arte_url(@arte, client_id: 99999)
  assert_redirected_to admin_artes_url
  assert Arte.exists?(@arte.id), "arte must not be deleted when guard fires"
end
```

### IN-03: `set_arte` raises an unhandled `RecordNotFound` when a valid `client_id` is paired with an `arte_id` that belongs to a different client

**File:** `app/controllers/admin/artes_controller.rb:77`

**Issue:** When `@client` is set by `set_client`, `set_arte` uses `@client.artes.find(params[:id])`. If the `:id` in the URL identifies an arte owned by a different client, `find` raises `ActiveRecord::RecordNotFound`. Rails catches this at the middleware level and renders a 404 page in production, but no friendly redirect or flash message is shown. This is purely an admin-facing UX concern (no security impact — access is correctly blocked) but it is inconsistent with the CR-01 redirect pattern applied elsewhere.

**Fix:** Rescue the lookup or use `find_by` with an explicit check:

```ruby
def set_arte
  scope = @client ? @client.artes : Arte
  @arte = scope.includes(:approval_responses).find_by(id: params[:id])
  unless @arte
    redirect_to admin_artes_path, alert: "Arte não encontrada." and return
  end
  @client ||= @arte.client
end
```
