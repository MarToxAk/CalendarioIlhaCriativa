<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL /c/:token → Client lookup | Token no URL não é segredo (enviado via WhatsApp) — senha é a segunda camada |
| POST /c/:token/session → authenticate | Senha do cliente atravessa esta fronteira |
| session cookie → require_client_auth | Cookie contém client_id + client_token_version |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Elevation of Privilege | Cross-client access via URL manipulation | mitigate | load_client_from_token usa o token do URL para lookup; require_client_auth verifica session[:client_id] contra @client.id — token de outro cliente não autentica sessão atual |
| T-05-02 | Elevation of Privilege | Session fixation após rotação de token | mitigate | session[:client_token_version] verificado a cada request em require_client_auth; regenerate_access_token muda token_version imediatamente |
| T-05-03 | Spoofing | Timing attack na autenticação de senha | mitigate | `@client.authenticate(password)` usa bcrypt#==(password) que é timing-safe por design; nunca comparar digests manualmente |
| T-05-04 | Information Disclosure | Link único do cliente enviado por WhatsApp | accept | Token de 58^24 possibilidades torna força bruta inviável; Rack::Attack throttle no GET /c/:token (Plano 04) limita enumeração |
| T-05-05 | Tampering | params[:token] como discriminator no Rack::Attack | mitigate | O throttle extrai token do PATH (req.path) — não de params do body — imune a manipulação do body |
| T-05-06 | Spoofing | Acesso ao ClientController via admin concern | mitigate | `skip_before_action :require_authentication` no ClientController — os dois fluxos de auth são completamente separados |

</threat_model>
