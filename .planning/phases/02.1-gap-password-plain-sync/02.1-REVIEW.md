# Phase 02.1: Code Review Report

**Reviewed:** 2026-05-27
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Reviewed the `Admin::ClientsController` and its test suite, which implement the `password_plain` sync feature. The controller correctly handles the common case — syncing `password_plain` whenever `password` is submitted — but permits `password_plain` as a direct parameter in strong params, creating a desync vector. The test suite covers the primary happy paths but lacks coverage for the `rotate_token` action, unauthenticated access, and flash message assertions on the `update` paths.

---

## Critical Issues

### CR-01: `password_plain` Writable Independently of `password` via Direct Param Submission

**File:** `app/controllers/admin/clients_controller.rb:69`

**Issue:** `client_params` permits `:password_plain` as a standalone parameter. In the `update` action, the guard that syncs `password_plain` from `password` only fires when `filtered[:password].present?`. If a crafted PATCH request submits `client[password_plain]=arbitrary_value` with no `password` key (or with `password` blank), the reject block removes the blank `password` but leaves the non-blank `password_plain` untouched, and the sync guard never fires. The result is `password_plain` being written to the database with a value that does not match `password_digest`. The displayed portal password on the admin show page becomes incorrect, and any admin tool that reads `password_plain` to communicate the credential to the client would deliver a wrong value.

The same path theoretically applies to `create`, but `has_secure_password` validation (`password can't be blank`) will reject the record on create when no `password` is supplied, so the impact is limited to `update`.

**Reproduction path:**
```
PATCH /admin/clients/:id
params: { client: { password_plain: "hacked_display_value" } }
# password key absent → reject block ignores password_plain (not blank)
# filtered[:password].present? → false → no merge
# password_plain stored as "hacked_display_value", password_digest unchanged
```

**Fix:** Remove `:password_plain` from the permitted parameters entirely. `password_plain` should only ever be written by controller code, never from user input. Let the controller control the column explicitly:

```ruby
def client_params
  params.require(:client).permit(:name, :password, :active)
end
```

Then in both `create` and `update`, write `password_plain` only when `password` is present and valid — which the existing merge logic already does. Removing the permit entry closes the desync vector without changing any observable behavior for legitimate requests.

## Warnings

### WR-01: No Test for `rotate_token` Action

**File:** `test/controllers/admin/clients_controller_test.rb`

**Issue:** The `rotate_token` action (line 56-59 of the controller) has zero test coverage. It calls `@client.regenerate_access_token` and sets a flash notice. The action depends on `set_client` (the `before_action`) and requires authentication. None of this is verified. A regression in `regenerate_access_token`, routing, or the before-action chain would go undetected.

**Fix:** Add at minimum one integration test:

```ruby
test "rotate_token regenera o token e redireciona com notice" do
  token_anterior = @client.access_token
  post rotate_token_admin_client_path(@client)
  assert_redirected_to admin_client_path(@client)
  assert_match "Token rotacionado", flash[:notice]
  @client.reload
  assert_not_equal token_anterior, @client.access_token
end
```

### WR-02: No Test for Unauthenticated Access to Any Action

**File:** `test/controllers/admin/clients_controller_test.rb`

**Issue:** Every test in the suite calls `sign_in_as(@admin)` in `setup`, so every test runs authenticated. There is no test verifying that unauthenticated requests are redirected to the login page. If `require_authentication` were accidentally removed from `Admin::BaseController` (or the `before_action` chain broken), no test would catch the regression and all admin client data would become publicly accessible.

**Fix:** Add at least one test per verb or one representative test that signs out first:

```ruby
test "requer autenticacao — redireciona para login sem sessao" do
  sign_out
  get admin_clients_path
  assert_redirected_to new_session_path
end
```

### WR-03: `update` Tests for CLIE-03 Do Not Assert Flash Notice Content

**File:** `test/controllers/admin/clients_controller_test.rb:76-88`

**Issue:** The deactivation and reactivation tests assert `assert_redirected_to` and `@client.active?` but do not verify the branch-specific flash messages (`"foi desativado"` / `"foi reativado"`). The controller has three distinct notice branches in `update`. If the wrong branch fires (e.g., due to an off-by-one in the `was_active` comparison), the model state assertion alone would pass while the wrong message was shown to the user.

**Fix:**

```ruby
test "update com active: false desativa o cliente (CLIE-03)" do
  patch admin_client_path(@client), params: { client: { active: false } }
  assert_redirected_to admin_client_path(@client)
  assert_match "foi desativado", flash[:notice]
  @client.reload
  assert_equal false, @client.active?
end

test "update com active: true reativa o cliente (CLIE-03)" do
  @client.update!(active: false)
  patch admin_client_path(@client), params: { client: { active: true } }
  assert_redirected_to admin_client_path(@client)
  assert_match "foi reativado", flash[:notice]
  @client.reload
  assert_equal true, @client.active?
end
```

## Info

### IN-01: UI Placeholder Claims "Minimum 4 Characters" but No Model Validation Enforces It

**File:** `app/views/admin/clients/_form.html.erb:28`

**Issue:** The password field placeholder reads `"Mínimo 4 caracteres"`, creating a user expectation of a minimum length. The `Client` model uses `has_secure_password` with no `minimum:` option, so Rails only validates presence (on create) and the BCrypt maximum of 72 characters. A 1-character password would pass model validation and be saved. The UI claim is misleading.

**Fix:** Either add a model validation to match the stated constraint:

```ruby
has_secure_password
validates :password, length: { minimum: 4 }, allow_nil: true
```

(`allow_nil: true` preserves the update-without-password-change behavior.) Or update the placeholder to remove the minimum length claim.

### IN-02: CLIE-04 Test Does Not Assert Flash Notice

**File:** `test/controllers/admin/clients_controller_test.rb:58-65`

**Issue:** The test `"update com nova senha sincroniza password_plain (CLIE-04)"` asserts `assert_response :redirect` (implicit) and `@client.reload` state, but does not assert `flash[:notice]`. The controller sends `"Dados do cliente atualizados."` on the generic update path. While this is a minor gap, it leaves the generic notice message branch untested.

**Fix:**

```ruby
assert_equal "Dados do cliente atualizados.", flash[:notice]
```
