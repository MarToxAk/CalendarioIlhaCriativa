# Phase 03.1: Code Review Report

**Reviewed:** 2026-05-27T00:00:00Z
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

This phase implements the arte creation flow from the client show page, including the arte form partial, the cancel/redirect logic, the client show page art listing, and associated controller/test changes. The implementation is largely functional, but two security-class issues were found — one XSS vector via `raw(body)` with unescaped user-controlled data in the confirm modal, and one design-level security concern with plaintext password storage surfaced through the show view. Four additional warnings cover a nil-crash risk on `scheduled_on`, a logically incorrect sort in the approval history, a hidden form state leak on cancel, and an incomplete guard in the delete check. Three info items cover test gaps and dead nav items.

---

## Critical Issues

### CR-01: XSS via `raw(body)` with unescaped `@client.name` in confirm modal

**File:** `app/views/admin/clients/show.html.erb:25` and `app/views/admin/clients/show.html.erb:104`

**Issue:** The `body:` string passed to the `confirm_modal` partial is built with Ruby string interpolation (`"#{@client.name}"`) and then rendered with `raw(body)` inside `_confirm_modal.html.erb` at line 34. Rails auto-escaping is bypassed by `raw()`. If a client record is created with a name containing HTML/JavaScript — e.g. `<img src=x onerror=alert(1)>` — the payload executes in the admin's browser when they visit the client show page. An admin account takeover via stored XSS is the realistic attack scenario; the admin panel has no CSP in evidence.

**Fix:** Pass the name through `html_escape` (aliased `h`) before embedding it into an HTML-containing string:

```ruby
# In show.html.erb, line 25:
body: "Desativar <strong>#{h(@client.name)}</strong> bloqueará o acesso ao portal imediatamente. O cliente não conseguirá acessar o calendário até ser reativado.",

# In show.html.erb, line 104:
body: "&#9888; <strong>Atenção</strong><br><br>Ao rotacionar o token, o link atual de <strong>#{h(@client.name)}</strong> para de funcionar imediatamente ...",
```

Alternatively, refactor `_confirm_modal.html.erb` to accept a separate `client_name` local and render it with `<%= %>` (auto-escaped) instead of building an HTML string on the caller side.

### CR-02: Plaintext password rendered in admin view — `password_plain` stored in DB

**File:** `app/views/admin/clients/show.html.erb:64`

**Issue:** `@client.password_plain` is a real database column (`db/schema.rb:80: t.string "password_plain"`) storing the client's portal password in cleartext. This column is rendered directly into the page and also passed to a clipboard copy helper. This is a systemic security design defect: a single database breach exposes every client's portal credential in plain form. The concern is surfaced here because this phase's show page explicitly displays and copies the value.

**Fix:** The correct long-term fix is to remove `password_plain` from the schema and derive a shareable credential differently (e.g., generate a single-use setup link, or show the password only once at creation time without persisting it). As a short-term hardening measure, at minimum restrict database-level access to this column and audit who can query `clients.password_plain`. This is an architectural issue that predates this phase but is directly exercised by the added show-page rendering.

## Warnings

### WR-01: `arte.scheduled_on.strftime` will crash if `scheduled_on` is nil

**File:** `app/views/admin/clients/show.html.erb:153` and `app/views/admin/clients/show.html.erb:181`

**Issue:** Both the "Artes" card and the "Histórico de aprovações" card call `arte.scheduled_on.strftime(...)` without a nil guard. The model validates `scheduled_on: presence: true`, but the validation only fires on new records; existing records with a null `scheduled_on` (e.g., created by a migration, direct DB insert, or a future schema change) will raise `NoMethodError: undefined method 'strftime' for nil` and crash the show page for that client.

**Fix:**
```ruby
<%= arte.scheduled_on&.strftime("%d/%m/%Y") || "—" %>
```

Apply at lines 153 and 181.

### WR-02: `sort_by(&:created_at)` reverses the association's default `ORDER BY` and is redundant/incorrect

**File:** `app/views/admin/clients/show.html.erb:183`

**Issue:** The `Arte` model declares `has_many :approval_responses, -> { order(created_at: :desc) }`. The association therefore returns responses newest-first. The view then does `arte.approval_responses.sort_by(&:created_at).last` — `sort_by` sorts ascending by default, so `.last` returns the newest record. This accidentally gives the correct result *only* because `sort_by` re-sorts ascending, making `.last` the newest. However, this is fragile: it loads all responses into memory in desc order, reverses them in Ruby via `sort_by`, then takes the last — three redundant operations. More importantly, if a developer reads the code, the combination of desc association + ascending sort + `.last` is confusing and may be changed incorrectly. The direct intent is "get the most recent response."

**Fix:**
```ruby
<% last_response = arte.approval_responses.first %>
```

The association is already ordered `desc`, so `.first` is the newest. This is clear, correct, and avoids the Ruby-level sort entirely.

### WR-03: Cancel link on new arte form drops `client_id` context — user lands on global artes list, not back on client page

**File:** `app/views/admin/artes/_form.html.erb:45`

**Issue:** When a new arte is being created from the client show page (via "Nova Arte" link which passes `client_id` as a query param), the cancel link resolves to `admin_artes_path` (the global artes index). The user arrived from `admin_client_path(@client)`, so the expected cancel destination is back to the client's show page. This breaks the navigation flow. The `arte` object carries `arte.client` (set in the controller via `Arte.new(client: @client)`) but the form ignores it for the cancel target.

**Fix:**
```erb
<%= link_to "Cancelar",
      arte.persisted? ? admin_arte_path(arte) :
        (arte.client_id? ? admin_client_path(arte.client_id) : admin_artes_path),
      class: "btn" %>
```

### WR-04: `check_deletable` guard queries `approval_responses` on the already-loaded `@arte` but bypasses the preloaded association

**File:** `app/controllers/admin/artes_controller.rb:77`

**Issue:** `set_arte` loads `@arte` with `.includes(:approval_responses)`, but `check_deletable` calls `@arte.approval_responses.none?` — Active Record's `.none?` fires a new `COUNT(*)` SQL query instead of using the preloaded array. This is not strictly a bug (it returns the correct answer) but it does mean the guard is silently issuing an extra query every time `destroy` is called. More critically, if a caller ever changes `set_arte` to not include approval_responses, the guard will still work by issuing another query — making the preload appear redundant and potentially removed. The guard should use `.empty?` on the loaded collection.

**Fix:**
```ruby
def check_deletable
  unless @arte.pending? && @arte.approval_responses.empty?
    redirect_to admin_arte_path(@arte), alert: "Exclusão bloqueada: só é possível excluir artes pendentes sem resposta do cliente."
  end
end
```

`.empty?` on an already-loaded association uses the in-memory array instead of issuing a new query.

## Info

### IN-01: Test sends `status: :pending` in create params but `status` is not in `arte_params` — parameter is silently ignored

**File:** `test/controllers/admin/artes_controller_test.rb:28`

**Issue:** The `should create arte` test POSTs `status: :pending` in the params hash. The controller's `arte_params` method does not permit `:status`, so Rails' strong parameters silently discards it. The test passes because `status` defaults to `pending` (enum integer 0) at the DB level, not because the status parameter is honoured. This masks the fact that the test is not testing what it claims, and could mislead a developer into thinking status can be set via the create endpoint.

**Fix:** Remove `status: :pending` from the create test params. If testing that `status` cannot be mass-assigned is desired, add an explicit test that posts `status: :approved` and asserts the created record has `pending` status.

### IN-02: Sidebar nav items with `path: "#"` will all highlight as "active" if the browser URL is literally `/#`

**File:** `app/views/admin/shared/_sidebar.html.erb:12-18`

**Issue:** Four nav items use `path: "#"` as placeholder routes. `current_page?("#")` evaluates to `true` whenever the request path is `/` concatenated with the anchor fragment. In practice this is unlikely because browsers do not send the fragment to the server, but the intent of `current_page?` against `"#"` is undefined/unreliable. When the real routes are implemented, these stubs should be replaced.

**Fix:** Replace `"#"` paths with their real named routes when implemented. Until then, use `javascript:void(0)` or `nil` as the href and render `<span>` instead of `<a>` for unimplemented items to avoid misleading active-state styling.

### IN-03: Test suite has no coverage for create failure (validation error) or update success path

**File:** `test/controllers/admin/artes_controller_test.rb`

**Issue:** The test file covers the happy-path create and several status-guard scenarios, but there is no test for:
1. POST to `admin_artes_url` with missing required fields (e.g., no `client_id`, no media source) — should render `:new` with `422`.
2. PATCH to `admin_arte_url` on a pending arte with valid params — the standard update happy path is untested (only `update_admin_reply` which is a special case).

Without a failing-create test, regressions in validation rendering would not be caught.

**Fix:** Add:
```ruby
test "should not create arte without client" do
  assert_no_difference("Arte.count") do
    post admin_artes_url, params: { arte: { title: "X" } }
  end
  assert_response :unprocessable_entity
end

test "should update pending arte" do
  patch admin_arte_url(@arte), params: { arte: { title: "Novo Título" } }
  assert_redirected_to admin_arte_url(@arte)
  assert_equal "Novo Título", @arte.reload.title
end
```
